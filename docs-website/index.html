mongo-keepalive

Keep MongoDB Atlas free-tier clusters alive with periodic ping commands

npm Python Rust Go Java C# Ruby PHP Dart Kotlin

The Problem

MongoDB Atlas pauses free-tier (M0) clusters after 60 days of inactivity. When your cluster is paused, you'll need to manually resume it before you can connect again.

mongo-keepalive automatically sends periodic ping commands to keep your cluster active, ensuring it never gets paused.

Getting Started

Installation

npm install mongo-keepalive

Basic Usage

const { startKeepAlive } = require('mongo-keepalive');

// Start keep-alive with default 12-hour interval
const handle = await startKeepAlive('mongodb+srv://user:pass@cluster.mongodb.net/db');

// Or with custom interval
const handle = await startKeepAlive(
    'mongodb+srv://user:pass@cluster.mongodb.net/db',
    '6h'
);

// Stop when needed
handle.stop();

Configuration

Interval Formats

The interval parameter accepts various formats:

"5m"    // 5 minutes
"30m"   // 30 minutes
"1h"    // 1 hour
"6h"    // 6 hours (recommended)
"12h"   // 12 hours (default)
"24h"   // 24 hours
"1d"    // 1 day

Connection String

Use standard MongoDB connection strings:

// Standard format
mongodb://user:pass@host:27017/db

// MongoDB Atlas (SRV)
mongodb+srv://user:pass@cluster.mongodb.net/db

// Local development
mongodb://localhost:27017/local

Retry Logic

Automatic retry on failure:

  • Max retries: 3 attempts
  • Delay between retries: 5 seconds
  • Exponential backoff supported
  • Automatic reconnection on network failures

API Reference

startKeepAlive(uri, interval?)

Description: Initializes and starts the keep-alive process.

Parameters:

  • uri (string, required) - MongoDB connection string
  • interval (string, optional) - Ping interval (default: "12h")

Returns: Handle object with control methods

handle.stop()

Description: Gracefully stops the keep-alive process.

Parameters: None

Returns: Promise/Void (language dependent)

handle.status()

Description: Returns current status of keep-alive.

Parameters: None

Returns: Status object with active flag and last ping time

Error Handling

All implementations include comprehensive error handling:

  • Connection failures are logged and retried
  • Invalid intervals throw descriptive errors
  • Network timeouts are handled gracefully
  • All errors include timestamps and details

Frequently Asked Questions

Will this increase my MongoDB Atlas costs?

No. The ping command is extremely lightweight and doesn't count against your storage or compute quotas. Free-tier clusters remain free.

How often should I ping?

The default 12-hour interval is recommended. This is frequent enough to prevent inactivity while minimizing unnecessary operations.

Can I use this in production?

Yes! All implementations include proper error handling, logging, and retry logic suitable for production use.

Does this work with paid MongoDB Atlas tiers?

Yes, but it's primarily designed for free-tier (M0) clusters which auto-pause after 60 days of inactivity.

What happens if my internet connection drops?

The library will retry failed pings automatically and continue when connectivity is restored.

Can I stop the keep-alive process?

Yes, each implementation provides a stop() method to gracefully shutdown the background process.
